01 Revenue Operations

Stopping $450K/Month in Revenue Leakage

How I identified a silent billing gap between two systems and led the product solution that closed it.

Problem Statement

Finance flagged a recurring discrepancy between invoiced amounts and actual payments received. The gap was growing—$450K/month—but no one could pinpoint the source. It had been attributed to "system errors" for 8 months.

Context

The company ran a SaaS billing platform integrated with Stripe and a legacy CRM. Subscription upgrades, downgrades, and cancellations triggered events across both systems—but the reconciliation logic was manual and inconsistent.

My Role

Lead Business Analyst. I owned the investigation, stakeholder alignment, requirements definition, and UAT. I worked with Finance, Engineering, and the CRM team.

Approach

  1. Data deep-dive: Wrote SQL queries to cross-reference CRM subscription events against Stripe payment records. Found 3 event types that weren't triggering billing updates.
  2. Root cause mapping: Facilitated a 2-hour workshop with Engineering and Finance to map the full event flow. Identified the exact API webhook that was silently failing on plan downgrades.
  3. Stakeholder alignment: Presented findings to the CFO and CPO with a clear impact quantification. Got sign-off on a dedicated sprint.
  4. Requirements definition: Wrote detailed BRD and user stories covering the reconciliation engine, alerting logic, and audit trail requirements.
  5. UAT & validation: Designed test scenarios covering all 3 failure modes. Ran UAT with Finance team before go-live.

Solution

An automated reconciliation engine that runs nightly, flags discrepancies in real-time, and triggers corrective billing events. Included a Finance dashboard for visibility and a Slack alert for anomalies above $1K.

Impact

$450K/moRevenue recovered
100%Reconciliation accuracy
8 weeksDiscovery to production

Learnings

Silent failures are the most expensive kind. The fix was straightforward—the hard part was proving the problem existed. This reinforced my belief that BAs need SQL fluency, not just process skills. Data is the fastest path to stakeholder buy-in.

02 Product Revamp

Revamping the Accommodation Booking Product

How a 30% conversion lift came from asking "why do users leave?" before writing a single requirement.

Problem Statement

The accommodation booking funnel had a 68% drop-off rate between property selection and payment confirmation. The product team had been shipping UI tweaks for 6 months with no measurable improvement.

Context

A travel-tech platform with 200K monthly active users. The booking flow spanned 6 steps across web and mobile. Previous attempts focused on visual redesigns without addressing underlying friction points.

My Role

Senior BA embedded in the product squad. I led discovery, defined the problem space, wrote requirements, and coordinated delivery across UX, Engineering, and Data teams.

Approach

  1. Funnel analysis: Partnered with Data to build a step-by-step drop-off analysis. Identified 4 specific steps with disproportionate abandonment.
  2. User research: Ran 12 moderated user sessions. Uncovered 3 recurring pain points: unclear pricing, confusing cancellation policy placement, and a broken promo code flow on mobile.
  3. Prioritization: Used an impact/effort matrix to rank 14 identified issues. Focused the first release on the top 5 highest-impact, lowest-effort fixes.
  4. Requirements: Wrote detailed user stories with acceptance criteria for each fix. Included edge cases for mobile-specific behaviors.
  5. Iterative delivery: Shipped in 3 two-week sprints. Monitored conversion metrics after each sprint to validate impact before proceeding.

Solution

Restructured pricing transparency (showing total cost earlier), moved cancellation policy to a persistent sidebar, and rebuilt the promo code component for mobile. Also added a booking summary step before payment.

Impact

+30%Conversion rate
-42%Support tickets on booking
6 weeksTo first measurable lift

Learnings

The previous team was solving the wrong problem. UI polish doesn't fix friction—understanding user intent does. Discovery is not a phase; it's a discipline. I now insist on at least 5 user sessions before writing a single requirement.

03 API Integration

Partner API Integration at Scale

How I bridged the gap between business requirements and technical architecture to ship 3 partner integrations simultaneously.

Problem Statement

The business needed to onboard 3 strategic partners via API within a single quarter to unlock a new revenue stream. Each partner had different data schemas, authentication methods, and SLA requirements.

Context

A marketplace platform expanding its supplier network. The engineering team had capacity for one integration at a time. Business stakeholders wanted all three live simultaneously for a joint marketing launch.

My Role

Lead BA and integration coordinator. I owned the requirements for all 3 integrations, managed partner communication, and designed the phased delivery approach that made parallel shipping possible.

Approach

  1. Partner discovery: Ran structured discovery sessions with each partner's technical team. Documented API specs, data models, error handling, and rate limits.
  2. Abstraction layer design: Worked with the architect to define a common integration interface. This meant each partner integration could be built in parallel against the same internal contract.
  3. Requirements per partner: Wrote integration specs covering authentication flows, data mapping, error scenarios, retry logic, and monitoring requirements.
  4. Risk mapping: Identified 6 high-risk dependencies (partner API stability, data schema mismatches). Created mitigation plans for each.
  5. Parallel delivery: Coordinated 3 engineering sub-teams working simultaneously. Ran weekly sync with all partners and internal stakeholders.

Solution

A modular integration framework with a shared adapter layer, per-partner configuration, and a unified monitoring dashboard. All 3 partners went live within the same 2-week window.

Impact

3 partnersLaunched simultaneously
On timeJoint marketing launch hit
ReusableFramework for future partners

Learnings

The abstraction layer idea came from a 30-minute whiteboard session with the architect. BAs who understand technical architecture unlock solutions that pure process thinkers miss. I've since made "architecture alignment" a standard step in my integration discovery process.
